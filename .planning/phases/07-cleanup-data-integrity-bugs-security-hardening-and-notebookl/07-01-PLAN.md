<objective>
Tier 1 — Data integrity bug fixes for the infographic generation pipeline.

Three locked decisions from CONTEXT.md, all surgical edits to two route files:

1. **D-01 — Claim field drift.** The current routes read `c.type`, `c.amount`, and `c.filedDate` from each claim. The Prisma `InsuranceClaim` model exposes `claimType`, `approvedValue`, and `dateOfLoss`. Every claim field currently lands as `undefined` in the NotebookLM payload, which means customer briefings are silently missing real claim data.

2. **D-02 — Fake-sequential batch loop.** The batch route's loop comment claims "Fire processing sequentially (NotebookLM has rate limits)" but launches each call via `.then()/.catch()` fire-and-forget. The result is N parallel jobs hitting a single shared headless-browser session, which trips rate limits and corrupts state. Make it truly sequential by `await`-ing each call inside the loop, wrapping each iteration in try/catch so a single failure does not abort the batch.

3. **D-03 — Silent first-user fallback.** Both routes resolve `userId` and, if `session.user.id` does not match a real DB user, fall back to `prisma.user.findFirst()` (or `customer.assignedRepId`). This is dead code from earlier dev-bypass days that, if ever triggered, silently misattributes work to a random user. The current `auth.ts` uses `PrismaAdapter`, so a session always points to a real user — the fallback is impossible to hit legitimately. Remove it; if `session.user.id` does not exist in the `User` table, return 401.

This plan is Wave 1 because data correctness blocks the security and operational hardening that follows in plans 07-02 and 07-03 (those plans modify the same files for additional concerns). No new features, no refactoring, no new dependencies — just surgical edits to two files.

Purpose: Stop silently corrupting NotebookLM input data, restore strict user attribution, and make NotebookLM's rate-limit-sensitive backend actually receive sequential calls.
Output: Two route files with corrected field names, true sequential processing, and strict 401-on-unknown-user behavior. No new tests (the codebase explicitly excludes `src/app/api/**` from coverage in `vitest.config.ts:20` — verification is grep + manual smoke per CONTEXT.md discretion clause).
